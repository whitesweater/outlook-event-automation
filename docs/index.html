Outlook / Gmail -> AI -> Calendar

把活动邮件变成可信的日历事件

这是一个常驻服务器上的邮件处理代理：它读取你的活动邮件，用结构化 AI 抽取时间、地点和标题， 只把足够确定的事件写入日历，并把原始邮件内容附在日历详情里，方便之后追溯。

给用户你只需要把活动邮件送进目标邮箱，服务会自动识别、过滤、写入日历。
给开发者核心是 Microsoft Graph/Gmail API + Responses API + SQLite 去重 + systemd 常驻。
保守写入多活动汇总、取消通知、时间冲突、信息不完整，不会自动写进日历。
可追溯日历事件正文包含 AI 摘要、来源邮件元数据和原始邮件正文。

给用户看的

它到底帮你做什么

这个部分关心使用结果：哪些邮件会进日历、哪些不会、日历项里有什么。

自动处理活动邮件

讲座、工作坊、报名活动、培训、维护通知等邮件会被识别，提取标题、开始时间、结束时间、地点和说明。

不把垃圾通知塞进日历

账号登录通知、天气提醒、普通系统通知会被标记为 ignored，不会进入复核队列，也不会写入日历。

遇到不确定就停住

一封邮件里有多个活动、时间冲突、缺少结束时间、取消通知或撤回通知，会进入 needs_review，不会自动创建。

日历项能追溯来源

Outlook Calendar 事件正文会附上活动摘要、原始邮件主题、发件人、收信时间、原始链接和邮件正文。

默认中文化

标题、摘要、地点、复核原因尽量用中文写；英文项目名、机构名、房间名会尽量保留。

常驻运行

服务部署在服务器上，以 systemd 常驻；新邮件进入后按轮询间隔处理，不依赖本地电脑一直开着。

给开发者看的

系统是怎么工作的

这个部分关心实现：API、模型、状态机、去重、批处理和部署边界。

1. 读取消息

Outlook 通过 Microsoft Graph 读取 `/me/messages` 或指定 mailbox；Gmail 通过 Gmail API 读取邮件正文。

2. 前置规则过滤

Daily Event Alert 汇总邮件直接忽略；取消和撤回类邮件会被强制复核，不允许自动创建。

3. 批量 AI 抽取

默认每批 20 封邮件，请求 OpenAI-compatible Responses API，使用 JSON schema 要求每封邮件返回一个事件判断。

4. 自动拆批重试

如果代理 500、上游 reset、或模型返回数量不一致，批次会自动从 20 拆成 10、5，单封失败不写 processed，留给下一轮重试。

5. 写入与记录

eligible 事件写入 Outlook/Google Calendar；SQLite 记录 source message id、dedupe key、状态和远端 event id。

状态规则

为什么不是什么都自动写

日历是高信任界面，宁愿少写，也不要把错误事件塞进去。

created

时间、地点、标题、置信度都足够，且不是重复事件。服务模式下会创建日历项。

dry_run

测试模式下的可写候选，只展示结果，不真的写日历。

needs_review

像活动，但缺结束时间、多活动混杂、信息冲突、取消/撤回，暂时只保留日志。

ignored

明确不是日程，或者命中特定噪声规则，例如 Daily Event Alert 汇总邮件。

部署方式

从仓库到服务器

项目没有强依赖框架，核心脚本是标准库 Python，适合放在小服务器上长期运行。

运行前需要什么

你需要一个 Microsoft 或 Google OAuth 应用、一个 OpenAI-compatible Responses API key，以及一份本地配置文件。

生产环境不要提交 .env、config.local.json、SQLite 数据库或扫描日志。

Quickstart 
cd outlook_event_automation
cp config.example.json config.local.json
cp .env.example .env

python3 event_agent.py --config config.local.json auth-microsoft
python3 event_agent.py --config config.local.json run \
  --source outlook --sink none --limit 20 --force
python3 event_agent.py --config config.local.json serve --write
